🏗️ ชั้นที่ 1
Hierarchy

จัดลำดับชั้น ระเบียบ → แนวปฏิบัติ → แบบฟอร์ม ตอบคำถาม "มีเอกสารที่เกี่ยวข้องอะไรบ้าง" แสดง doc_link ครบถ้วนทุก children

🔗 ชั้นที่ 2
Cross-Reference

เชื่อมระดับข้อระหว่างเอกสาร รองรับข้าม KB ตอบคำถาม "ข้อนี้ต้องปฏิบัติยังไง" พร้อมอธิบายเชิงลึก

หลักการ: ทั้งสองชั้นทำงานอิสระ — Hierarchy ตอบ "รายการเอกสาร" Cross-Reference ตอบ "เนื้อหาเชิงลึก" Orchestrator เลือกชั้นตาม intent
คำถามชั้นที่ใช้ผลลัพธ์
ระเบียบ PDPA มีเอกสารอะไรบ้าง?Hierarchyรายการ children + doc_link ครบ
ข้อ 5 ของระเบียบ PDPA ต้องทำอะไร?Cross-Refเนื้อหา + articles ที่เกี่ยวข้อง
สรุปสิ่งที่ต้องทำเรื่อง PDPA ทั้งหมดCombinedHierarchy หา scope + Cross-ref อธิบาย
ค้นหาเอกสารเรื่องข้อมูลส่วนบุคคลDirect RAGChunk ที่เกี่ยวข้องจาก KB
📊 ระดับชั้นเอกสาร (Hierarchy Levels)
L1
ระเบียบ / กฎกระทรวง / มาตรฐาน
เอกสารหลักระดับสูงสุด มีสถานะผูกพัน — parent_doc_id = null
REG-PDPA-001 · STD-ISO37301 · RULE-CYBER-001
L2
แนวปฏิบัติ / คู่มือ / ประกาศ
รายละเอียดวิธีปฏิบัติ อยู่ใต้ระเบียบ L1
GUIDE-PDPA-001 · MANUAL-CYBER-001 · ANNOUNCE-DPO-001
L3
แบบฟอร์ม / คำสั่ง / Template
เครื่องมือสนับสนุน อยู่ใต้แนวปฏิบัติ L2
FORM-CONSENT-001 · ORDER-DPO-001 · TMPL-REPORT-001
L4
FAQ / ตัวอย่าง / Q&A
เอกสารเสริมความเข้าใจ แขวนใต้ L2 หรือ L3
FAQ-PDPA-001 · EXAMPLE-CYBER-001
⚠️ กฎสำคัญ: เมื่อเพิ่มเอกสารใหม่ ต้องอัปเดต children array ของ Parent ทันที หากลืม Orchestrator จะตอบว่า "ไม่มีเอกสารเกี่ยวข้อง"

📝 YAML Frontmatter มาตรฐาน Hybrid

Level 1 — ระเบียบ / กฎกระทรวง (Root)

yaml — REG-PDPA-001
---
# ── ข้อมูลพื้นฐาน ────────────────────────────────
doc_id:          "REG-PDPA-001"
doc_title:       "ระเบียบการคุ้มครองข้อมูลส่วนบุคคล กฟภ. 2566"
doc_type:        "ระเบียบ"
domain:          "KB_IT_CYBER"
effective_date:  "2566-01-01"
doc_link:        "https://drive.google.com/file/d/FILE_ID/view"
tags:            ["PDPA","ข้อมูลส่วนบุคคล","DPO","ความยินยอม"]

# ── ชั้นที่ 1: HIERARCHY ──────────────────────────
hierarchy:
  level:          1
  parent_doc_id:  null                # null = Root ไม่มี Parent
  children:
    - doc_id:     "GUIDE-PDPA-001"
      doc_title:  "แนวปฏิบัติการคุ้มครองข้อมูลส่วนบุคคล กฟภ."
      doc_link:   "https://drive.google.com/file/d/GUIDE_ID/view"
      kb:         "KB_IT_CYBER"
      level:      2
    - doc_id:     "ANNOUNCE-DPO-001"
      doc_title:  "ประกาศแต่งตั้งเจ้าหน้าที่คุ้มครองข้อมูล (DPO)"
      doc_link:   "https://drive.google.com/file/d/DPO_ID/view"
      kb:         "KB_IT_CYBER"
      level:      2
  siblings:       []

# ── ชั้นที่ 2: CROSS-REFERENCE (ระดับข้อ) ──────────
cross_refs:
  - doc_id:       "GUIDE-PDPA-001"
    maps_to:      "ข้อ 5"
    articles:     [16,17,18,19,20]
    kb:           "KB_IT_CYBER"
  - doc_id:       "STD-ISO27001"
    maps_to:      "ข้อ 5"
    articles:     ["A.8.2"]
    kb:           "KB_COMPLIANCE"     # Cross-KB
---

🔗 Cross-Reference ในเนื้อหา Markdown

แบบที่ 1 — Internal Link

markdown
### ข้อ 5 สิทธิของเจ้าของข้อมูล
ผู้ให้บริการต้องแจ้งวัตถุประสงค์ก่อนเก็บข้อมูลตาม[ข้อ 7](#ข้อ-7-การขอความยินยอม)
การขอความยินยอมต้องทำตาม[แบบฟอร์มที่กำหนด](#ภาคผนวก-ก)

แบบที่ 2 — Cross-Document Reference

markdown
> **📎 ดูแนวปฏิบัติเพิ่มเติม:**
> 📄 [แนวปฏิบัติ PDPA ข้อ 16–20](https://drive.google.com/file/d/GUIDE_ID/view)
>
> <!-- XREF: doc_id=GUIDE-PDPA-001 kb=KB_IT_CYBER articles=16,17,18,19,20 maps_from=ข้อ5 -->

แบบที่ 3 — Cross-KB Reference (ข้าม Knowledge Base)

สำคัญ: ใช้เมื่อเอกสารใน KB_IT_CYBER อ้างอิงมาตรฐาน ISO ที่อยู่ใน KB_COMPLIANCE
markdown
> **📎 มาตรฐานสากลที่เกี่ยวข้อง:**
> 🌐 [ISO 27001 Annex A.8.2](https://drive.google.com/file/d/ISO_ID/view)
>
> <!-- XREF: doc_id=STD-ISO27001 kb=KB_COMPLIANCE articles=A.8.2 maps_from=ข้อ5 -->
> <!-- XREF: doc_id=REG-PDPA-NATIONAL kb=KB_COMPLIANCE articles=26,27,28 maps_from=ข้อ5 -->
🔄 Auto-Detection: Orchestrator เพิ่ม KB_COMPLIANCE อัตโนมัติเมื่อ intent=procedure + domain=KB_IT_CYBER หรือคำถามมี "ISO/มาตรฐาน/ขั้นตอน"
📂 Format ต่างประเภท
Formatวิธีจัดการdoc_typeหมายเหตุ
PDF ข้อความCopy-paste → เพิ่ม YAML + XREFระเบียบ/แนวปฏิบัติวิธีมาตรฐาน
PDF สแกนOCR → แก้ตัวสะกด → เพิ่ม YAMLระเบียบตรวจ OCR errors ก่อน
Word (.docx)Export → Markdown → เพิ่ม YAMLแนวปฏิบัติ/คู่มือใช้ Pandoc
กฎกระทรวงระบุ "มาตรา" แทน "ข้อ" ใน RegexกฎกระทรวงPattern: มาตรา\s+\d+
แบบฟอร์มMarkdown อธิบายวิธีกรอก + doc_linkแบบฟอร์มไม่ต้อง OCR แบบฟอร์มจริง
🤖 Prompt สร้าง Markdown

Prompt แปลงเอกสารเป็น Markdown

prompt
แปลงเอกสารกฎระเบียบไทยต่อไปนี้เป็น Markdown สำหรับระบบ RAG โดย:
1. ใช้ ## สำหรับหมวด, ### สำหรับข้อ, #### สำหรับวรรค
2. รักษาหมายเลขข้อและหมวดตามเอกสารต้นฉบับทุกตัว
3. ใช้ > blockquote + HTML comment สำหรับ cross-reference ทุกแห่ง
4. รูปแบบ: <!-- XREF: doc_id=XXXX kb=KB_NAME articles=N,N maps_from=ข้อN -->
5. ใช้ bullet points (-) สำหรับรายการ
6. ไม่ต้องเพิ่ม YAML frontmatter

เนื้อหา: [วางเนื้อหาที่นี่]

Prompt สร้าง YAML Frontmatter

prompt
สร้าง YAML Frontmatter แบบ Hybrid Structure สำหรับเอกสาร กฟภ. นี้:

ข้อมูล:
- doc_id: [กรอก]
- parent_doc_id: [กรอก หรือ null]
- level: [1/2/3/4]
- domain: [KB_IT_CYBER/KB_HR_PROC/KB_OPERATIONS/KB_COMPLIANCE]
- doc_link: [Google Drive link]

จากเนื้อหา Markdown ให้:
1. สร้าง tags (5-8 คำ)
2. ระบุ cross_refs ทุก XREF comment ที่พบ
3. กำหนด doc_type ที่ถูกต้อง
4. ใส่ children: [] และ siblings: []
