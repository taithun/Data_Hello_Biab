What Hello Biab ทำได้ — 3 ระดับ Benefit

LEVEL 1 — ความเร็ว
ประหยัดเวลา 18 นาที/session
ข้ามขั้นตอนค้นหาไฟล์ + เปิด PDF + Ctrl+F ได้เลย นี่คือ Benefit ที่วัดเป็นตัวเลขในโมเดล
LEVEL 2 — ความครบถ้วน
ตอบได้ครบ ไม่พลาดข้อที่เกี่ยวข้อง
ระบบเดิม "หาเจอแต่ไม่ครบ" เพราะ Ctrl+F หาได้แค่คำที่ค้น — HyDE+Rerank ดึง Chunks ที่เกี่ยวข้องทั้งหมดมาให้ ลด Compliance Risk
LEVEL 3 — Unmet Demand
ตอบ 60% ที่ระบบเดิมตอบไม่ได้
~29 คน/วันที่หาไฟล์ไม่เจอ คือ Demand ที่รอการตอบสนองอยู่แล้ว ไม่ต้องสร้างใหม่ — Adoption Rate 2x จึง Conservative มาก

เหตุใด Adoption Rate จริงอาจสูงกว่า 2x

// GA4 Baseline = 49 sessions/วัน (คนที่ใช้ระบบออนไลน์)
engaged_users = 49 × 40% = ~20 คน/วัน ← Download สำเร็จ
unmet_demand = 49 × 60% = ~29 คน/วัน ← หาไม่เจอ = โอกาสของ Hello Biab

// ถ้า Hello Biab Embed ใน CSAS/Intranet → ดักจับทั้ง 2 กลุ่ม
potential_users = engaged + unmet = ~49 คน/วัน จากระบบเดิม
// + Hidden demand (Desktop/Drive/ถามเพื่อน) ที่ไม่ผ่านระบบออนไลน์เลย
real_potential = ~3-5x ของ GA4 baseline = ~150-250 คน/วัน
ข้อสรุป: Realistic Scenario ที่ใช้ 200 คน/วัน จึงเป็นตัวเลขที่สมเหตุสมผลและป้องกันการโต้แย้งได้ดี — ไม่ต้อง Justify 2x แต่อธิบายได้ว่ามาจาก Unmet Demand ที่มีอยู่แล้ว

token usage & api cost breakdown

ราคาอ้างอิง: Claude Sonnet $3/M input tokens, $15/M output tokens · Rerank model $2/1,000 queries · Embedding (Large) $0.13/M tokens · อัตราแลกเปลี่ยน USD/THB = 37

Hello Biab Query Pipeline — 5 Steps

1
HyDE Generation
รับ User Query → สร้าง Hypothetical Document เพื่อ Improve Embedding Quality
Input: ~200 tokens (query + HyDE prompt) | Output: ~300 tokens → ใช้ Claude Sonnet (Large)
Input: 200 × $3/M = $0.0006 | Output: 300 × $15/M = $0.0045 | รวม $0.00105 = ~0.039 ฿
4% ของ cost
2
Embedding (Large Model)
Embed HyDE Document เป็น Vector สำหรับ Semantic Search ใน KB
~300 tokens | ค่า Embedding ถูกกว่า LLM มาก
300 × $0.13/M = $0.000039 = ~0.001 ฿ (เล็กน้อยมาก)
1% ของ cost
3
Knowledge Retrieval
Vector Search ใน KB → ดึง Top-K Chunks ที่เกี่ยวข้อง
Top-K = 5-8 chunks × ~500 tokens/chunk = ~3,000 tokens retrieved | ค่า Vector Search แทบไม่มี
ค่า Retrieval = ~$0 (Vector DB operation cost) — Chunks นำไปใช้ใน Step 4 และ 5
~0%
4
Rerank
จัดลำดับ Chunks ใหม่ด้วย Cross-Encoder Model → เพิ่ม Precision ของผลลัพธ์
Input: ~3,000 tokens (ทุก Chunk) | Rerank Model แยกต่างหาก
$2 per 1,000 queries = $0.002/query = ~0.074 ฿ | หรือถ้าคิดแบบ Token: ~0.222 ฿
22% ของ cost
5
Answer LLM (Large Model)
ประมวล Chunks + Query → ตอบพร้อมอ้างอิง หมวด/ข้อ + Google Drive Link (+ Case Study ในอนาคต)
Input: ~3,500 tokens (Chunks + Query + System Prompt) | Output: ~600 tokens → Claude Sonnet
Input: 3,500 × $3/M = $0.0105 | Output: 600 × $15/M = $0.009 | รวม $0.0195 = ~0.722 ฿
73% ของ cost

cost-benefit scenarios

วิธีคิด Benefit: Benefit = เวลาที่ประหยัดได้ × ต้นทุนค่าแรงต่อนาที × จำนวน Sessions ต่อปี โดย Benefit เป็น "Opportunity Cost" ไม่ใช่ Cash Saving — เวลาที่ประหยัดได้ต้องถูกนำไปใช้ในงานที่สร้างมูลค่าเพิ่ม

// เวลาที่ประหยัดได้ต่อ Session
saved_min = 22 (manual avg) - 4 (AI avg) = 18 นาที/session

// Benefit รวมต่อปี
benefit = users/วัน × 240 วัน × saved_per_session

// B/C Ratio และ ROI
BC_ratio = benefit ÷ total_ai_cost
ROI = (benefit - total_ai_cost) ÷ total_ai_cost × 100

สูตรคำนวณ B/C Ratio และ ROI ฉบับเต็ม

// === ฝั่ง BENEFIT ===
cost_per_min = salary ÷ (work_days × hrs_per_day × 60)
             = 35,000 ÷ (240 × 8 × 60) = 3.038 ฿/นาที

benefit_per_session = (manual_min - ai_min) × cost_per_min
                    = (22 - 4) × 3.038 = 54.68 ฿/session

annual_benefit = users_per_day × work_days × benefit_per_session

// === ฝั่ง COST ===
api_cost = users_per_day × work_days × cost_per_query
total_cost = api_cost + infra_annual

// === B/C และ ROI ===
BC_ratio = annual_benefit ÷ total_cost
ROI = (annual_benefit - total_cost) ÷ total_cost × 100 // %
payback_months = total_cost ÷ annual_benefit × 12
